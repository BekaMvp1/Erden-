# Рефакторинг производственного процесса

## Обнаруженные проблемы

1. **Порядок этапов в boardStages** — warehouse шёл до cutting; planning отсутствовал.
2. **Раскрой без планирования** — можно было создавать задачи раскроя без плана.
3. **Пошив без ограничения по раскрою** — можно было передать в пошив больше, чем раскроено.
4. **Факт пошива > план** — не было проверки.
5. **Planning UI** — пошаговый выбор «модель → даты → этаж», дневная таблица, лишние формы.
6. **Мощность** — жёсткая валидация 1000–5000, отсутствие overload в API.
7. **Планирование зависело от раскроя** — calc-capacity требовал факт раскроя.
8. **Sewing UI** — не отображалось available_for_sewing.

## Текущая логика процесса

```
Заказ → Планирование → Раскрой → Пошив → ОТК → Склад → Отгрузка
```

1. **Планирование** — резервирование мощности (production_plan_day: order_id, floor_id, date, planned_qty)
2. **Раскрой** — только для запланированных заказов; факт в actual_variants
3. **Пошив** — fact_sum ≤ раскроено - уже_в_пошиве; fact_sum ≤ plan_sum
4. **Завершение пошива** — создаётся SewingBatch со статусом READY_FOR_QC
5. **ОТК** — загружает партии READY_FOR_QC (batches/pending-qc)
6. **Сохранение ОТК** — партия переходит в QC_DONE, остатки на складе (warehouse_stock)

## Проверенные сценарии

- **Сценарий 1**: Создать заказ → Планирование → Раскрой (завершить по факту) → Пошив (завершить) → партия в ОТК ✓
- **Сценарий 2**: Раскроить 300 → в пошив можно передать ≤300; после пошива партия в ОТК ✓
- **Сценарий 3**: ОТК → склад (warehouse_stock) ✓

## Статусы SewingBatch

- `IN_PROGRESS` — в работе
- `READY_FOR_QC` — создаётся при «Завершить пошив»; партия в списке ОТК
- `QC_DONE` — после сохранения ОТК
- `DONE` — legacy
